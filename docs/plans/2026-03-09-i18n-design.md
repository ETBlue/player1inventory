# i18n Design: EN + TW Support

## Overview

Add internationalization support to Player 1 Inventory, starting with English (EN) and Traditional Chinese (TW). Users can view the app in either language. The app auto-detects browser language with a manual override in Settings.

## Scope

- **Languages:** EN (English), TW (Traditional Chinese / 繁體中文)
- **What gets translated:** UI text (labels, buttons, dialogs, placeholders, status messages), date/number formats, locale-appropriate default tag types on first launch
- **What stays as typed:** User-entered data (item names, tag names, vendor names, recipe names)
- **Deferred:** Japanese (JP)

## Architecture

### Library

- `react-i18next` + `i18next` — translation hook and core engine
- `i18next-browser-languagedetector` — automatic browser language detection

### File Structure

```
src/
  i18n/
    index.ts          # i18next initialization
    locales/
      en.json         # English strings
      tw.json         # Traditional Chinese strings
```

i18next initializes once at app entry (`src/main.tsx`). Single flat namespace to start; namespaces can be added later if files grow large.

**Fallback:** Missing keys in `tw.json` automatically fall back to `en.json`. Partial translation is safe.

**Usage in components:**
```tsx
const { t } = useTranslation()
<Button>{t('common.cancel')}</Button>
```

## Language Detection & Settings UI

### Detection Order

1. `localStorage` key `i18n-language` (user's explicit choice)
2. `navigator.language` — `zh-TW`, `zh-HK`, `zh` → `tw`; `en-*` → `en`; unknown → `en`

### Hook

`useLanguage()` in `src/hooks/useLanguage.ts` — mirrors `useTheme()` pattern:

```ts
const { preference, language, setPreference } = useLanguage()
// preference: 'auto' | 'en' | 'tw'
// language: 'en' | 'tw'  (resolved)
```

### Settings UI

New Language row in `/settings`, same pattern as Theme selector:

```
Language
[Auto (Browser) ▾]
Auto-detected: 繁體中文
```

Dropdown options: `Auto (Browser)`, `English`, `繁體中文`

- Selecting "Auto" clears `localStorage` preference; description shows which language was detected
- Selecting a specific language stores `'en'` or `'tw'` in `localStorage`

## Date & Number Formatting

### Dates — `src/lib/formatDate.ts`

Uses native `Intl.DateTimeFormat`:

```ts
// en: "Mar 9, 2026"
// tw: "2026年3月9日"
formatDate(date: Date, language: 'en' | 'tw'): string
```

### Relative Time — `src/lib/formatRelativeTime.ts`

Uses native `Intl.RelativeTimeFormat`:

```ts
// en: "3 days ago", "yesterday"
// tw: "3 天前", "昨天"
formatRelativeTime(date: Date, language: 'en' | 'tw'): string
```

Automatically picks the appropriate unit (days → weeks → months → years).

### Numbers & Quantities

No special formatting needed — quantities are numeric and display correctly cross-locale.

### Pluralization

i18next built-in plural keys replace inline ternary patterns:

```json
// en.json
"cart.packs": "{{count}} pack",
"cart.packs_other": "{{count}} packs"

// tw.json
"cart.packs": "{{count}} 包"
```

## Default Tag Types on First Launch

Seeding runs once on empty database, using the resolved language at first launch. Changing language later does not alter existing tags.

**EN defaults:**
- Storage (freezer, fridge, pantry)
- Diet (plant-based, low-GI, gluten-free)
- Category (produce, dairy, meat, grains, snacks, beverages)

**TW defaults:**
- 保存方式 (冷凍, 冷藏, 常溫)
- 飲食型態 (蔬食, 低GI, 無麩質)
- 食材分類 (蔬果, 乳製品, 肉, 穀物, 零食, 飲料)

## Implementation Strategy

**Option B — Infrastructure first, then page-by-page:**

1. **PR 1 (infrastructure):** Install libraries, create `src/i18n/index.ts`, create `en.json` + `tw.json` stubs, implement `useLanguage()` hook, add Language selector to Settings page, implement `formatDate()` and `formatRelativeTime()` utilities, update database seed for locale-aware defaults
2. **Subsequent PRs (page migrations):** Migrate pages one at a time — Shopping → Pantry → Cooking → Settings → Item Detail → etc.

Each migration PR extracts hardcoded strings from one page/component group into translation files. Untranslated pages automatically fall back to English during migration.

## Testing

### Unit Tests

- `formatDate()` — EN and TW output for today, a past date, a future date
- `formatRelativeTime()` — EN and TW output for yesterday, 3 days ago, 2 weeks ago
- `useLanguage()` — detection order: localStorage → browser → fallback to EN
- Translation key parity — assert `en.json` and `tw.json` have the same top-level keys (CI regression guard)

### Integration Tests

- Settings page — user can switch language; preference persists across reload
- Shopping page — renders key strings in TW when language is `tw`

### E2E

Not in scope for this phase.
