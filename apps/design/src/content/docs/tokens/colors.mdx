The color system uses **OKLCH** for all token values, providing perceptually uniform color representation.
All hue colors are normalized to the same lightness level (L=65% light / L=75% dark) to guarantee WCAG AA contrast across all 10 hues without per-color tuning.

The tokens are defined in `apps/web/src/design-tokens/theme.css` and imported directly into the design guide — no copy-paste.

## Token categories

| Category | Purpose |
|----------|---------|
| **Background** | Surface elevation hierarchy (base → surface → elevated) |
| **Foreground** | Text hierarchy (emphasized → default → muted) |
| **Accessory** | Borders and decorative elements |
| **Importance** | Semantic CTA colors (primary, secondary, destructive, neutral) |
| **Status** | Feedback states (ok, warning, error, inactive) |
| **Tag** | Tag colors — 10 hues with solid and inverse (tint) variants |

## Usage

Tokens are consumed through Tailwind utility classes via `@theme inline` mappings.

```tsx
// Background layers
<div className="bg-background-base">      {/* Page background */}
<div className="bg-background-surface">   {/* Toolbar / header */}
<div className="bg-background-elevated">  {/* Card / list item */}

// Importance colors — base for fills, foreground for colored text on neutral bg
<div className="bg-importance-primary" />                         {/* badge / button fill */}
<span className="text-importance-primary-foreground" />           {/* colored label on page */}
<div className="border-importance-destructive text-importance-destructive-foreground" />

// Status feedback
<div className="bg-status-ok-inverse text-status-ok-foreground" />

// Hue colors (tags)
<Badge className="bg-tag-teal text-tag-teal-inverse" />
<Badge className="bg-tag-teal-inverse border-tag-teal-accessory text-tag-teal-foreground" />
```

## When to use each group

| You need to... | Reach for... |
|----------------|-------------|
| Set a page background | `background-base` |
| Set a toolbar or header background | `background-surface` |
| Set a card or list item background | `background-elevated` |
| Color text (body copy, labels, headings) | `foreground-muted`, `foreground-default`, `foreground-emphasized` |
| Color borders and dividers | `accessory-muted`, `accessory-default`, `accessory-emphasized` |
| Fill a badge or button with an importance color | `importance-primary`, `importance-destructive`, etc. |
| Show a colored label or text on a neutral background | `importance-primary-foreground`, `importance-destructive-foreground`, etc. |
| Show an inventory status (ok / warning / error / inactive) | `status-ok`, `status-warning`, etc. for fills; `-foreground` for text on neutral bg; `-inverse` for section backgrounds |
| Color a tag badge | `tag-orange`, `tag-teal`, `tag-blue`, etc. |

**Background group** — three elevation layers for visual hierarchy without shadows. Base is the page; surface is for toolbars, sticky headers, and popovers; elevated is for cards and list items.

**Foreground group** — muted for metadata and placeholder text, default for body copy, emphasized for headings and high-contrast labels. Avoid hardcoding `text-white` or `text-black` — use `foreground-colorless-inverse` and `foreground-colorless` instead.

**Accessory group** — borders, dividers, and decorative rules. Use muted for subtle separators, default for standard borders, emphasized for interactive or prominent borders.

**Importance group** — two independent use cases, not a pairing:
- Base (`importance-primary`, `importance-destructive`, etc.) — fill color for badges, buttons, and indicators
- Foreground (`importance-primary-foreground`, etc.) — colored text placed on a neutral page background, e.g. a section label or status note

**Status group** — same two-use-case pattern as importance:
- Base — fill for icons, progress bars, and compact indicators
- Foreground — colored text on neutral backgrounds
- Inverse — tinted section or card background for status-aware layouts

**Tag group** — exclusively for tag badge colors. Ten hues cover the full spectrum so each tag type can have a distinct color. Use inverse (tint) variants for most badges; solid variants for emphasis or active state.

## Semantic vs primitive

Always reach for **semantic tokens** when building components. Semantic tokens carry intent — they adapt to light/dark mode automatically and communicate meaning (e.g. "this is a primary action" or "this item is expiring").

```tsx
// Correct — semantic intent, adapts to theme
<button className="bg-importance-primary text-foreground-colorless-inverse" />

// Wrong — raw tag hue value used on a button; carries no semantic meaning
<button className="bg-tag-teal" />
```

**The one exception is tag badges.** Tag tokens (`orange`, `teal`, `blue`, etc.) are the correct primitive choice for tag color assignment because the purpose _is_ to assign a visual hue, not to express a semantic state. Use the full set of tag variants (`base`, `-inverse`, `-foreground`, `-accessory`) to style the badge itself.

```tsx
// Correct — tag color tokens used intentionally for tag colorization
<Badge className="bg-tag-teal-inverse border-tag-teal-accessory text-tag-teal-foreground" />
<Badge className="bg-tag-teal" />  // solid variant for active/selected
```

Never use raw OKLCH values or Tailwind's built-in color scale (e.g. `bg-blue-500`) — those bypass the token system and will not respond to theme changes.

## Dark mode

All tokens are defined in both `:root` (light) and `.dark` (dark) selectors.
The theme is controlled by the `dark` class on `<html>` — toggled via `localStorage` and OS preference detection.
